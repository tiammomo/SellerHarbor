# SellerHarbor 开发路线图

SellerHarbor 已完成从“内容生成工具”到“跨境卖家商品运营港”的定位切换。后续路线图围绕商品主档、平台 Listing、海外仓库存、真实反馈、热款追踪和长期可靠性推进。

## 当前基线

- 产品名、仓库名、Docker 服务名、环境变量和 API Header 已统一为 SellerHarbor。
- 默认 Docker 栈包含 backend、frontend、MinIO，全部带健康检查。
- 首页和 Dashboard 已围绕商品、平台、仓库、库存和热款信号组织。
- 已新增默认店铺注册接口，当前单店铺可用，未来多店铺按 storeId 扩展。
- 内容与评价生成保留为子模块，继续遵守真实反馈、人工审核和风险拦截边界。
- Open Food Facts 和泛选品 Provider 不再是默认主线，只作为隐藏实验能力保留。
- 后端测试覆盖商品、反馈整理、生成、审核、租户、限流、指标和 readiness。

## Phase 1: 商品主档与运营驾驶舱

目标：让个人卖家打开系统就知道哪些商品、平台、仓库和热款需要处理。

范围：

- 稳定 `/api/commerce/overview`。
- 商品页作为商品主档入口。
- 基于商品属性展示 SKU、平台状态、仓库、库存、销量、评分和评价数。
- 首页展示上架就绪率、库存健康率、低库存预警、热款/潜力款和下一步动作。
- 剔除默认公开采集入口，避免实验接口影响首页。

验收：

- 首页和 Dashboard 无需 LLM 配置也可正常使用。
- 低库存、缺 Listing、热款库存风险都有可见动作。
- 核心概览接口和前端页面纳入端到端测试。

## Phase 2: 结构化库存与仓库管理

目标：减少共享海外仓导致的超卖、断货和补货滞后。

范围：

- 新增 Warehouse、Inventory、InventoryMovement、PlatformAllocation。
- 从 `products.attributes` 迁移库存字段。
- 支持安全库存、占用库存、在途库存和库存流水。
- 热款库存保护和补货建议。
- 库存变更审计。

验收：

- 可按仓库查看 SKU 库存和平台占用。
- 可识别低库存、断货和热款补货风险。
- 数据迁移脚本可重复执行并有回滚策略。

## Phase 3: 店铺、平台 Listing 与批量导入导出

目标：让 Temu / TikTok Shop 上架资料有统一维护入口。

范围：

- 新增 Store，支持默认单店铺迁移到结构化店铺主档。
- 新增 PlatformListing。
- PlatformListing 必须包含 storeId，支持同一商品对应多个平台店铺。
- Temu / TikTok Shop 字段完整度检查。
- CSV 导入导出。
- 批量编辑标题、卖点、图片、规格和平台状态。
- Listing 审核失败原因和待处理动作。

验收：

- 每个 SKU 可维护多个平台 Listing。
- 每个 Listing 可追溯到明确店铺。
- 缺失字段和待优化项可追踪。
- CSV 导入失败有清晰错误报告。

## Phase 4: 异步任务与高可用底座

目标：让平台同步、批量生成和数据整理在真实运营中可持续运行。

范围：

- PostgreSQL 替换 SQLite 作为主数据存储。
- Redis-backed worker 承接批量生成、平台同步和采集任务。
- 重试、幂等、死信队列和任务审计。
- 备份、恢复演练和迁移脚本。
- 集中日志、指标、告警和运行手册。

验收：

- 后端可横向扩展到多实例。
- 长任务重启后不会丢失状态。
- 备份可恢复到可运行环境。

## Phase 5: 权限、团队协作与平台同步

目标：支持真实商家团队协作和逐步自动化。

范围：

- 用户登录、角色和权限。
- 商家租户管理。
- 审核流和操作日志。
- Temu / TikTok Shop CSV / API 同步适配器。
- Temu 先完成只读同步，再推进写回能力；详见 [Temu 接入计划](TEMU_INTEGRATION.md)。
- 同步前预览、同步后审计和失败回滚。

验收：

- 不同商家数据隔离。
- 不同角色权限清晰。
- 平台同步可追踪、可重试、可回滚。

## 长期不做或延后

- 不做刷评、刷单、伪造身份或规避平台风控。
- 不在 MVP 阶段自动发布商品或自动调整平台库存。
- 不替代完整 ERP、财务、采购或平台官方履约系统。
- 不把公开商品库采集作为默认主线能力。
