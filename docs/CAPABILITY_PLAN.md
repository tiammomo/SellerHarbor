# SellerHarbor 长期能力规划

SellerHarbor 的长期定位是“跨境卖家的商品运营港”：把商品主档、Temu / TikTok Shop 上架状态、共享海外仓库存、评价反馈、热款信号和运营动作收口到一个可靠工作台。

## 核心主线

### 1. 商品主数据

- SKU / SPU / 变体统一建模。
- 平台字段映射：Temu、TikTok Shop 优先。
- 商品资料完整度：标题、类目、卖点、图片、规格、禁用表达。
- CSV 导入导出和批量编辑。

### 2. 平台 Listing 管理

- 每个 SKU 对应多个平台 Listing。
- Listing 状态：未创建、草稿、审核中、已上架、被拒、暂停。
- 平台字段完整度检查。
- 上架素材准备：标题、描述、图片清单、详情页卖点、邀评/客服话术。

### 3. 海外仓与库存

- 仓库主档：国家、地址、服务平台、时效说明。
- SKU 仓库存量：可用、占用、安全库存、在途、损耗。
- 库存流水：入库、出库、调整、平台占用释放。
- 低库存、断货和热款库存保护。

### 4. 热款与评价追踪

- 销量、评分、评价数、反馈摘要和库存状态综合打分。
- 热款、潜力款、风险款、滞销款分类。
- 差评关键词和客服反馈归因。
- 补货、扩平台、补素材和风险复核建议。

### 5. 内容与评价运营

- 作为子模块保留。
- 继续支持真实反馈整理、评价邀请、客服回访、推荐语、详情页口碑素材。
- 坚持人工审核、来源追踪和风险拦截。

## 高可用与长期维护

- 所有核心能力必须有端到端测试。
- 默认页面只依赖核心接口，实验接口不得影响首页可用性。
- 后台任务必须支持重试、幂等、超时恢复和审计。
- 生产数据需要备份、恢复演练、迁移脚本和租户隔离。
- 平台同步先从 CSV / 手工导入开始，API 自动化必须在审计和回滚能力就绪后再上线。

### 运行底座路线

- Docker 默认栈只保留长期运行组件：backend、frontend、MinIO，保证本地重启后没有无意义的退出容器。
- MinIO 作为当前对象资产底座，必须有健康检查；backend 必须等待对象存储可用后再启动。
- 当前 SQLite 是 MVP 持久化方案，需要本地备份安全网；生产高可用前必须迁移到 PostgreSQL。
- Redis 先作为可启动基础设施保留，等采集、平台同步、批量生成成为核心流程后再接入 worker、重试和死信队列。
- 高可用推进顺序：健康检查与启动依赖 -> 备份与恢复演练 -> PostgreSQL 迁移 -> Redis worker -> 反向代理/TLS -> 集中日志、指标和告警。

## 延后能力

- 自动发布商品。
- 自动调整平台库存。
- 完整 ERP / 财务 / 采购系统。
- 复杂 CRM、营销自动化和会员运营。
- 多平台广告投放自动化。

## 剔除或降级能力

- 公开商品库低频采集作为默认功能剔除，不再出现在主导航和全局加载链路。
- Open Food Facts 仅保留为隐藏实验接口，不作为 SellerHarbor 的核心能力。
- 泛选品 Provider 框架从 MVP 主线移除；后续若恢复，必须服务于“已有商品主档的外部验证”，而不是独立做选品平台。
- 首页和商品主档不再展示外部采集链路，除非该商品已有明确的外部验证报告。
- “口碑运营”不再作为顶层战略，而是收敛为内容与评价运营子模块。

## 阶段路线

### Phase 1: 商品主档与驾驶舱

- 稳定 `/api/commerce/overview`。
- 商品页改为商品主档。
- 首页展示商品、平台、仓库、库存预警和热款动作。
- 剔除默认公开采集入口。

### Phase 2: 结构化库存

- 新增 Warehouse、Inventory、InventoryMovement、PlatformAllocation。
- 从商品 attributes 迁移到独立库存表。
- 补低库存、断货、热款补货测试。

### Phase 3: 平台 Listing

- 新增 PlatformListing。
- Temu / TikTok Shop 字段完整度检查。
- CSV 导入导出和批量编辑。

### Phase 4: 同步与可靠性

- 平台 CSV / API 适配器。
- 异步任务队列、重试、死信和审计。
- 备份恢复、告警、权限和团队协作。
